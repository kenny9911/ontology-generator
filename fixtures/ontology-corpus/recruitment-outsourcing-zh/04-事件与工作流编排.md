# 事件与工作流编排 —— 外包招聘全流程 (RAAS)

**版本:** 3.1
**归属部门:** 招聘技术平台组 (Recruitment Tech Platform) 与交付运营中心 (DOC)
**生效日期:** 2026-01-01
**关联文档:** `01-业务流程SOP.md` (RAAS-SOP-001)、`02-业务规则与制度.md` (RAAS-POL-002)、`03-系统与数据实体.md` (RAAS-SYS-003)。

## 1. 事件目录 (Event Catalog)

下表列出每个事件: 名称、载荷字段、由哪个动作产生 (生产者)、被哪个动作消费 (消费者)。

### E-01 `需求已提交 (RequestSubmitted)`
- **载荷:** `client_id`, `contact_id`, `submitted_at`, `position_brief`
- **由动作产生:** 客户在 ClientPortal 提交需求 (外部触发)
- **被动作消费:** `CreateRecruitmentRequest`

### E-02 `需求已创建 (RequestCreated)`
- **载荷:** `request_id`, `client_id`, `consultant_id`, `headcount`
- **由动作产生:** `CreateRecruitmentRequest`
- **被动作消费:** `AlignRequirement`

### E-03 `需求已对齐 (RequirementAligned)`
- **载荷:** `request_id`, `position_id`, `salary_band_id`, `required_skills`
- **由动作产生:** `AlignRequirement`
- **被动作消费:** `ValidateContractTerms`

### E-04 `合同条款已校验 (ContractTermsValidated)`
- **载荷:** `request_id`, `contract_id`, `fee_rate`, `guarantee_days`
- **由动作产生:** `ValidateContractTerms`
- **被动作消费:** `GenerateJobDescription`

### E-05 `项目已暂挂 (RequestOnHold)`
- **载荷:** `request_id`, `reason`(枚举: `ContractExpired`/`Overdue`), `flagged_at`
- **由动作产生:** `ValidateContractTerms` (异常分支)
- **被动作消费:** 交付经理处理 (人工) / 监控工作流 WF-6

### E-06 `JD已生成 (JobDescriptionDrafted)`
- **载荷:** `jd_id`, `position_id`, `revision_count`
- **由动作产生:** `GenerateJobDescription`
- **被动作消费:** `ApproveJobDescription`

### E-07 `JD需修订 (JobDescriptionRevisionRequested)`
- **载荷:** `jd_id`, `revision_count`, `client_notes`
- **由动作产生:** `ApproveJobDescription` (修改分支)
- **被动作消费:** `GenerateJobDescription` (回环)

### E-08 `JD已确认 (JobDescriptionApproved)`
- **载荷:** `jd_id`, `position_id`, `approved_by`
- **由动作产生:** `ApproveJobDescription`
- **被动作消费:** `PublishToChannels`

### E-09 `JD已发布 (JobPostingPublished)`
- **载荷:** `posting_id`, `jd_id`, `channel_id`, `posted_at`
- **由动作产生:** `PublishToChannels`
- **被动作消费:** `IngestResume`

### E-10 `简历已采集 (ResumeIngested)`
- **载荷:** `resume_id`, `candidate_id`, `position_id`, `channel_id`
- **由动作产生:** `IngestResume`
- **被动作消费:** `ParseAndScoreResume` (MatchEngine)

### E-11 `匹配评分已生成 (MatchScored)`
- **载荷:** `application_id`, `overall_score`, `skill_match_pct`
- **由动作产生:** `ParseAndScoreResume`
- **被动作消费:** `FilterCandidatePool`

### E-12 `候选池已筛选 (CandidatePoolFiltered)`
- **载荷:** `position_id`, `to_screen_count`, `not_matched_count`
- **由动作产生:** `FilterCandidatePool`
- **被动作消费:** `ConductPhoneScreen`

### E-13 `初筛已完成 (ScreeningCompleted)`
- **载荷:** `application_id`, `result`, `expected_salary`, `available_date`
- **由动作产生:** `ConductPhoneScreen`
- **被动作消费:** `ScheduleInterview`

### E-14 `面试已安排 (InterviewScheduled)`
- **载荷:** `interview_id`, `application_id`, `interviewer_id`, `round`, `scheduled_at`
- **由动作产生:** `ScheduleInterview`
- **被动作消费:** `SubmitAssessmentReport`

### E-15 `面试需改期 (InterviewRescheduleRequested)`
- **载荷:** `interview_id`, `application_id`, `no_show_count`
- **由动作产生:** `ScheduleInterview` (异常分支)
- **被动作消费:** `ScheduleInterview` (回环) / 流失置位

### E-16 `评估已提交 (AssessmentSubmitted)`
- **载荷:** `report_id`, `interview_id`, `rating`, `advance_decision`
- **由动作产生:** `SubmitAssessmentReport`
- **被动作消费:** `ConsolidateFeedback`

### E-17 `面试反馈已汇总 (FeedbackConsolidated)`
- **载荷:** `application_id`, `final_status`(`NextRound`/`ToRecommend`/`Rejected`)
- **由动作产生:** `ConsolidateFeedback`
- **被动作消费:** `AssembleRecommendationPackage`

### E-18 `推荐包已交付 (RecommendationDelivered)`
- **载荷:** `package_id`, `position_id`, `candidate_ids`, `delivered_at`
- **由动作产生:** `AssembleRecommendationPackage`
- **被动作消费:** `RecordClientFeedback`

### E-19 `客户已决策 (ClientDecisionRecorded)`
- **载荷:** `package_id`, `client_decision`(`Hire`/`SecondRound`/`Reject`)
- **由动作产生:** `RecordClientFeedback`
- **被动作消费:** `NegotiateSalary` (录用分支) / `ScheduleInterview` (复试分支)

### E-20 `薪资已达成 (SalaryAgreed)`
- **载荷:** `negotiation_id`, `candidate_id`, `proposed_salary`, `guarantee_days`
- **由动作产生:** `NegotiateSalary`
- **被动作消费:** `IssueOffer`

### E-21 `Offer已发出 (OfferIssued)`
- **载荷:** `offer_id`, `candidate_id`, `annual_salary`, `start_date`
- **由动作产生:** `IssueOffer`
- **被动作消费:** `RunBackgroundCheck`、`RecordOfferResponse`

### E-22 `背调已完成 (BackgroundCheckCleared)`
- **载荷:** `check_id`, `offer_id`, `result`(`Clear`)
- **由动作产生:** `RunBackgroundCheck`
- **被动作消费:** `RecordOfferResponse`

### E-23 `背调异常 (BackgroundCheckFlagged)`
- **载荷:** `check_id`, `offer_id`, `flag_detail`
- **由动作产生:** `RunBackgroundCheck` (异常分支)
- **被动作消费:** 交付经理 / 客户复核 (人工), 可触发 Offer 撤回

### E-24 `Offer已接受 (OfferAccepted)`
- **载荷:** `offer_id`, `candidate_id`, `accepted_at`
- **由动作产生:** `RecordOfferResponse`
- **被动作消费:** `CreateOnboardingTicket`

### E-25 `Offer已拒绝 (OfferDeclined)`
- **载荷:** `offer_id`, `candidate_id`, `decline_reason`
- **由动作产生:** `RecordOfferResponse`
- **被动作消费:** `AssembleRecommendationPackage` (回环, 推荐备选)

### E-26 `入职单已创建 (OnboardingTicketCreated)`
- **载荷:** `ticket_id`, `offer_id`, `report_date`, `probation_days`
- **由动作产生:** `CreateOnboardingTicket`
- **被动作消费:** `ConfirmOnboarding`

### E-27 `候选人已入职 (CandidateOnboarded)`
- **载荷:** `ticket_id`, `candidate_id`, `report_date`
- **由动作产生:** `ConfirmOnboarding`
- **被动作消费:** `TriggerBillingMilestone`

### E-28 `回款里程碑已触发 (BillingMilestoneTriggered)`
- **载荷:** `milestone_id`, `fee_id`, `milestone_type`(`Onboard70`), `amount`
- **由动作产生:** `TriggerBillingMilestone`
- **被动作消费:** `SettleGuaranteePeriod`

### E-29 `服务费已结算 (ServiceFeeSettled)`
- **载荷:** `fee_id`, `total_settled`, `settled_at`
- **由动作产生:** `SettleGuaranteePeriod`
- **被动作消费:** 项目关闭 (人工 / 系统)

### E-30 `保证期内离职 (GuaranteeBreached)`
- **载荷:** `guarantee_id`, `ticket_id`, `candidate_id`, `resigned_at`
- **由动作产生:** `SettleGuaranteePeriod` (异常分支)
- **被动作消费:** `AssembleRecommendationPackage` (免费替补回环) / `TriggerBillingMilestone` (按比例返还)

### E-31 `SLA已超时 (SLABreached)`
- **载荷:** `entity_type`, `entity_id`, `sla_node`, `breached_at`, `escalate_to`
- **由动作产生:** TalentFlow ATS 定时监控 (任意超时节点)
- **被动作消费:** 监控工作流 WF-6 升级处理

## 2. 工作流定义 (Workflow Definitions)

下面定义 6 条端到端工作流, 将上述动作按顺序、分支与事件驱动串联。

---

### WF-1 —— 需求到发布工作流 (Request-to-Posting)
- **触发事件:** `需求已提交 (RequestSubmitted)`
- **参与角色:** 招聘顾问、客户对接人、财务专员、寻源专员
- **编排策略:** 顺序编排 + 客户审批回环 (saga)
- **动作链:**
  1. `CreateRecruitmentRequest` → 产生 `需求已创建`
  2. `AlignRequirement` → 产生 `需求已对齐`
  3. `ValidateContractTerms` → 产生 `合同条款已校验`
     - **分支 (合同失效 / 欠费):** 产生 `项目已暂挂`, 转 WF-6 升级, 暂停本流程
  4. `GenerateJobDescription` → 产生 `JD已生成`
  5. `ApproveJobDescription` → 产生 `JD已确认`
     - **分支 (需修改):** 产生 `JD需修订` → 回到步骤 4; 修订次数 > 3 时升级交付经理 (规则 R-08)
  6. `PublishToChannels` → 产生 `JD已发布`
- **结束态:** `JobPosting` 状态为【在线 (Live)】

---

### WF-2 —— 采集到推荐工作流 (Sourcing-to-Recommendation)
- **触发事件:** `JD已发布 (JobPostingPublished)`
- **参与角色:** 寻源专员、MatchEngine、招聘顾问、客户面试官
- **编排策略:** 事件驱动流式处理 (每份简历独立流转) + 评分门槛分支
- **动作链:**
  1. `IngestResume` (每份简历) → 产生 `简历已采集`
  2. `ParseAndScoreResume` (MatchEngine) → 产生 `匹配评分已生成`
  3. `FilterCandidatePool` → 产生 `候选池已筛选`
     - **分支条件:** `MatchScore ≥ 70` 进入【待初筛】, 否则【不匹配】终止 (规则 R-16)
  4. `ConductPhoneScreen` → 产生 `初筛已完成`
     - **分支 (初筛淘汰):** `Application` 置【Rejected】终止
  5. `ScheduleInterview` → 产生 `面试已安排`
     - **分支 (爽约):** 产生 `面试需改期`; 连续两次爽约置【流失】(规则 R-24)
  6. `SubmitAssessmentReport` → 产生 `评估已提交`
  7. `ConsolidateFeedback` → 产生 `面试反馈已汇总`
     - **分支:** `NextRound` 回到步骤 5; `ToRecommend` 继续; `Rejected` 终止
  8. `AssembleRecommendationPackage` → 产生 `推荐包已交付`
- **结束态:** `RecommendationPackage` 在 ClientPortal 发布 (需达到规则 R-23 的 2–5 名)

---

### WF-3 —— 客户反馈到 Offer 工作流 (Feedback-to-Offer)
- **触发事件:** `推荐包已交付 (RecommendationDelivered)`
- **参与角色:** 客户对接人、客户面试官、招聘顾问、薪酬顾问、交付经理、背调专员
- **编排策略:** 分支编排 (录用 / 复试 / 拒绝) + 审批门
- **动作链:**
  1. `RecordClientFeedback` → 产生 `客户已决策`
     - **分支 (复试 SecondRound):** 转 WF-2 步骤 5 安排额外面试轮次
     - **分支 (拒绝 Reject):** 转 WF-2 步骤 8 推荐备选 / 补充候选池
     - **分支 (录用 Hire):** 继续 (前置校验规则 R-22: No Hire 不得进入)
  2. `NegotiateSalary` → 产生 `薪资已达成`
     - **分支条件 (超预算上限):** `SalaryNegotiation` 置【Escalated】, 交付经理审批 (规则 R-21)
  3. `IssueOffer` → 产生 `Offer已发出` (前置: 客户 ClientPortal 批准, 规则 R-27; 服务费 > 80,000 需交付经理审批, 规则 R-04)
  4. `RunBackgroundCheck` (并行) → 产生 `背调已完成` 或 `背调异常`
     - **分支 (背调异常):** 重大不实则撤回 Offer (规则 R-26), 转 WF-2 推荐备选
  5. `RecordOfferResponse` → 产生 `Offer已接受` 或 `Offer已拒绝`
     - **分支 (拒绝 / 超 5 天未答复 Expired):** 规则 R-12, 转 WF-2 推荐备选
- **结束态:** `Offer` 状态为【已接受 (Accepted)】

---

### WF-4 —— 入职到回款工作流 (Onboarding-to-Billing)
- **触发事件:** `Offer已接受 (OfferAccepted)`
- **参与角色:** 招聘顾问、客户对接人、财务专员
- **编排策略:** 顺序编排 + 计时器驱动 (保证期)
- **动作链:**
  1. `CreateOnboardingTicket` → 产生 `入职单已创建`
  2. `ConfirmOnboarding` → 产生 `候选人已入职` (前置: 背调通过 + 入职材料齐备, 规则 R-28、R-34)
  3. `TriggerBillingMilestone` (Onboard70) → 产生 `回款里程碑已触发` (入职当日 70%, 规则 R-30)
  4. `SettleGuaranteePeriod` (保证期到期计时器, 默认 90 天) → 产生 `服务费已结算`
- **结束态:** `ServiceFee` 状态为【已结算 (Settled)】, 项目关闭

---

### WF-5 —— 保证期违约与替补工作流 (Guarantee-Breach & Replacement)
- **触发事件:** `保证期内离职 (GuaranteeBreached)`
- **参与角色:** 财务专员、招聘顾问、交付经理、客户对接人
- **编排策略:** 事件驱动 + 补偿事务 (compensation) 二选一分支
- **动作链:**
  1. `SettleGuaranteePeriod` 检测到保证期内离职 → 产生 `保证期内离职`
  2. **分支 A (免费替补):** `AssembleRecommendationPackage` 重新启动 WF-2/WF-3, 30 天内交付替补推荐包 (规则 R-32); 不重复收取首期服务费
  3. **分支 B (按比例返还):** `TriggerBillingMilestone` (Refund) 按未满保证期天数比例返还已收服务费 (规则 R-31), `ServiceFee` 置【Refunded】
- **结束态:** 替补候选人入职 (回到 WF-4) 或服务费返还完成

---

### WF-6 —— SLA 监控与升级工作流 (SLA Monitoring & Escalation)
- **触发事件:** `SLA已超时 (SLABreached)` 或 `项目已暂挂 (RequestOnHold)`
- **参与角色:** TalentFlow ATS (监控)、招聘顾问、交付经理、客户对接人
- **编排策略:** 横切监控 (cross-cutting) + 定时轮询 + 分级升级
- **动作链:**
  1. TalentFlow ATS 定时扫描各 SLA 节点 (需求→推荐 7 天、推荐→反馈 3 天、面试→评估 2 天、Offer→答复 5 天、背调 5 天) → 产生 `SLA已超时`
  2. 第一级升级: 通过 CommChannel 通知责任招聘顾问, 要求 24 小时内响应 (规则 R-14)
  3. **分支 (超时未响应):** 第二级升级至交付经理
  4. 对 `项目已暂挂`: 交付经理联系客户处理合同 / 回款, 处理完毕后恢复 WF-1
- **结束态:** SLA 节点恢复正常或升级闭环

## 3. 编排策略总览

| 工作流 | 触发事件 | 主要分支条件 | 编排策略 |
|---|---|---|---|
| WF-1 需求到发布 | `RequestSubmitted` | 合同失效 / JD 修订回环 | 顺序 + 审批回环 |
| WF-2 采集到推荐 | `JobPostingPublished` | 评分门槛 70 / 爽约 / 反馈分级 | 事件驱动流式 |
| WF-3 反馈到 Offer | `RecommendationDelivered` | 录用 / 复试 / 拒绝 / 超预算 / 背调异常 | 分支 + 审批门 |
| WF-4 入职到回款 | `OfferAccepted` | 材料齐备 / 背调通过 | 顺序 + 计时器 |
| WF-5 保证期替补 | `GuaranteeBreached` | 免费替补 vs 按比例返还 | 补偿事务 |
| WF-6 SLA 监控 | `SLABreached` / `RequestOnHold` | 分级升级 | 横切监控 + 轮询 |
